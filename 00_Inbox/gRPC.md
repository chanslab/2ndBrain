#RPC #REST #HTTP2
**gRPC(Google Remote Procedure Call)**는 구글에서 개발한 오픈소스 고성능 **RPC(원격 프로시저 호출)** 프레임워크입니다.

쉽게 설명하자면, 서로 다른 서버(또는 서비스)들이 마치 **자기 자신의 함수를 호출하는 것처럼 아주 빠르고 간편하게 데이터를 주고받을 수 있게 해주는 기술**입니다.

---

### 1. gRPC의 핵심 특징

gRPC가 기존의 REST API와 차별화되는 가장 큰 특징은 다음 세 가지입니다.

- **Protocol Buffers(Protobuf):** JSON 대신 이진(Binary) 형식의 Protobuf를 사용하여 데이터를 직렬화합니다. 메시지 크기가 훨씬 작고 파싱 속도가 압도적으로 빠릅니다.
    
- **HTTP/2 기반:** 최신 네트워크 프로토콜인 HTTP/2를 사용하여 양방향 스트리밍, 헤더 압축, 멀티플렉싱(하나의 연결로 여러 요청 처리) 등을 지원합니다.
    
- **IDL(인터페이스 정의 언어):** `.proto` 파일에 서비스 구조를 정의하면, 다양한 언어(Java, Go, Python, C++ 등)의 클라이언트/서버 코드를 자동으로 생성해 줍니다.
    

### 2. REST vs gRPC 비교

| **특징**      | **REST**   | **gRPC**             |
| ----------- | ---------- | -------------------- |
| **데이터 형식**  | JSON (텍스트) | Protobuf (이진)        |
| **프로토콜**    | HTTP/1.1   | HTTP/2               |
| **엄격함**     | 느슨함 (자율적)  | 엄격함 (계약 기반)          |
| **성능**      | 보통         | **매우 높음**            |
| **브라우저 지원** | 우수함        | 직접 호출 어려움 (Proxy 필요) |

---

### 3. gRPC의 주요 작동 방식

gRPC는 크게 4가지 통신 방식을 지원합니다.

1. **Unary RPC:** 일반적인 요청/응답 방식입니다.
    
2. **Server Streaming:** 클라이언트가 요청을 보내면 서버가 연속적인 데이터 스트림을 보냅니다. (예: 주식 시세 업데이트)
    
3. **Client Streaming:** 클라이언트가 스트림으로 데이터를 보내고 서버는 하나로 응답합니다. (예: 대용량 파일 업로드)
    
4. **Bidirectional Streaming:** 양쪽 모두가 동시에 스트림을 주고받습니다. (예: 채팅, 실시간 게임)
    

---

### 4. 언제 gRPC를 사용하나요?

- **마이크로서비스 아키텍처(MSA):** 서비스 간의 내부 통신이 빈번할 때 성능 저하를 막기 위해 주로 사용합니다.
    
- **실시간 서비스:** 저지연(Low-latency)이 중요한 실시간 데이터 대시보드나 스트리밍 서비스에 적합합니다.
    
- **다국어 환경:** 서버는 Java로, 클라이언트는 Go로 되어 있는 경우 자동으로 생성된 코드를 통해 통신 규약을 맞추기 쉽습니다.
    

> **참고:** 브라우저에서 직접 호출하는 프론트엔드 통신보다는, **서버와 서버 사이(Back-to-Back)** 통신에서 가장 강력한 힘을 발휘합니다.
